🔀 umami-proxy

Personvernbevarende reverse-proxy foran Umami analytics – bygget med Pingora i Rust.

Hva gjør applikasjonen?

umami-proxy er en reverse proxy som sitter mellom klientene (nettlesere) og en Umami-instans. Den utfører følgende oppgaver før forespørselen sendes videre:

  • Bot-filtrering – blokkerer kjente bot user-agents (HTTP 403).
  • Personvern / redaksjon – fjerner eller maskerer personopplysninger (fødselsnummer, e-post, IP-adresser, telefonnummer, NAV-identer, filstier, navn m.m.) fra request-body.
  • Felt-validering – trunkerer felt som overstiger 500 tegn.
  • Annotering – beriker forespørselen med proxy-versjon og K8s-appinformasjon (team, app-navn, ingress).
  • Geo-lokasjon – videresender x-client-city / x-client-region som Vercel-kompatible headere.
  • K8s-integrasjon – watcher på nais.io/v1alpha1 Application-ressurser for å mappe origin → app/team via en LRU-cache med prefix-trie.

Endepunkter

ALL /* :6191

Hoved-proxy. Alle forespørsler videresendes til upstream Umami-instansen (UMAMI_HOST:UMAMI_PORT). Request-body blir validert, redaksjonert og annotert underveis.

Prosesseringsflyt

  1. request_filter – Bot-sjekk (403 om bot), spawn K8s-watcher ved første request, ekstraher origin & geo-headere.
  2. request_body_filter – Buffer body → parse JSON/URL-encoded → validate (trunkering) → redact (personvern) → annotate (proxy-versjon, app-info).
  3. upstream_request_filter – Sett Host, Transfer-Encoding: Chunked, geo-headere (X-Vercel-IP-Country, X-Vercel-City), og eventuelt path-prefix.
  4. upstream_peer – Koble til Umami upstream (med TLS om UMAMI_SNI er satt).
  5. response_filter – Logg status.
  6. logging – Oppdater Prometheus-metrikker.

GET /is_alive :6969

Liveness-/readiness-probe. Returnerer 200 OK. Alle andre stier returnerer 404.

GET /metrics :9090

AppInfo (K8s-cache)

Hentet fra nais.io/v1alpha1 Application CRD. Lagres i en LRU-cache (maks 2000 oppføringer) med prefix-trie for raskt oppslag.

struct AppInfo {
    app_name: String,           // metadata.name
    namespace: String,          // metadata.namespace (team)
    ingress: String,            // spec.ingresses[0]
    creation_timestamp: String, // metadata.creationTimestamp
}

Forespørselskropp (Umami JSON-payload)

Proxyen aksepterer to innholdstyper:

application/json application/x-www-form-urlencoded

URL-encoded data konverteres til JSON:

{
  "events": [...],    // fra parameter "e"
  "api-key": "...",   // fra parameter "client"
  "payload": {
    "hostname": "www.example.com",  // brukes til K8s-oppslag
    ...
  }
}

Etter prosessering annoteres kroppen med:

{
  "proxyVersion": "umami-proxy-1.0.0",
  "event_properties": {
    "team": "mitt-team",
    "app": "min-app",
    "ingress": "https://min-app.nav.no",
    "hostname": "www.example.com",
    ...
  }
}

Proxy-kontekst (intern)

struct Ctx {
    request_body_buffer: Vec<u8>,   // Bufrer hele forespørselskroppen
    location: Option<Location>,      // Geo-data fra GCP LB-headere
    ingress: String,                  // Origin-header (uten scheme)
    proxy_start: Option<Instant>,    // Tidspunkt for forespørsel
}

struct Location {
    city: String,     // fra x-client-city
    country: String,  // fra x-client-region
}

Feilhåndtering

Proxy-feil (egendefinerte)

  • RequestContainsInvalidJson
  • JsonCoParseError
  • NoMatchingPeer
  • PrematureBodyEnd
  • FieldTooLong

Kategoriserte Pingora-feil

  • SslError – TLS-handshake, sertifikat
  • ConnectionError – Timeout, refused, no route
  • ClientDisconnectedError
  • UpstreamConnectionFailure
  • UntrackedError – alt annet
